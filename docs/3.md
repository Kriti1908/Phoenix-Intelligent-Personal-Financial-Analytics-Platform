## 1. Problem Statement

After registering and logging in, the dashboard showed:
- **FHS Gauge at 0** — no `financial_health_scores` rows existed in the database
- **Spending Pie Chart empty** — category distribution returned empty because RLS blocked service queries
- **Budget Bars empty** — no `budgets` rows existed, and even if they did, RLS would block them
- **"No data"** messages not shown — components rendered broken/blank charts with no fallback

### Root Causes

There were three distinct bugs working together:

#### Bug 1: Missing Seed Data
`seed.sql` seeded 50 transactions with categories, but never seeded:
- `financial_health_scores` — the FHS table had no initial row for the test user
- `budgets` — no budget rows existed for any category for the current month

Without this data, the dashboard would always show zeros even when everything else worked correctly.

#### Bug 2: RLS Policy UUID Cast Error (Critical)
`init.sql` defined Row-Level Security (RLS) policies like:

```sql
CREATE POLICY user_isolation ON transactions 
  USING (user_id = current_setting('app.current_user_id', true)::UUID);
```

The `true` flag (missing_ok) makes `current_setting` return an **empty string `''`** when the variable is not set (instead of raising an error). However, casting an empty string to UUID:

```sql
''::UUID  -- ERROR: invalid input syntax for type uuid: ""
```

This causes PostgreSQL to throw an exception. When the analytics service queries the database without setting `app.current_user_id` in the session (as was the case), the RLS policy evaluation crashes, and PostgreSQL returns **zero rows** to the caller (the error is suppressed at the policy evaluation level, effectively filtering all rows out).

Result: `_get_category_distribution()`, `_get_budget_status()`, `_get_recent_transactions()` and `_get_latest_fhs()` all returned empty results even though data existed.

#### Bug 3: No Empty-State UI
Dashboard components rendered Recharts charts or bare "No budgets set yet" text with no structure when data was empty. This resulted in:
- A blank/broken spending pie chart area  
- "Poor" FHS score at 0 (misleading)
- No guidance for new users

## 2. What Was Implemented

### 2.1 RLS Policy Fix (`infra/postgres/init.sql`)

**Pattern: Defensive Programming / Null Safety**

Changed the RLS policy USING clause to use `nullif()` to safely handle the empty string case:

```sql
-- BEFORE (crashes on empty string → UUID cast):
USING (user_id = current_setting('app.current_user_id', true)::UUID)

-- AFTER (safely returns NULL → no rows, no crash):
USING (user_id = nullif(current_setting('app.current_user_id', true), '')::UUID)
```

**How it works:**
1. `current_setting('app.current_user_id', true)` → returns `''` when not set
2. `nullif('', '')` → returns `NULL`
3. `NULL::UUID` → `NULL` (valid, no cast error)
4. `user_id = NULL` → evaluates to `NULL` (not TRUE), so the row is excluded
5. Result: unset session → no rows returned (correct isolation without crashing)

This fix is applied identically to the `transactions`, `financial_health_scores`, `anomaly_alerts`, and `budgets` tables.

> **Note**: This fix takes effect on a fresh database initialization. For running containers, run `docker compose down -v && docker compose up` to reset and re-initialize.

### 2.2 Per-Request RLS Context (`services/analytics/main.py`)

**Pattern: Repository Pattern**

Added `get_db_for_user(user_id)` — a factory function that returns an async generator yielding a DB session with the user's ID set as a PostgreSQL session-local variable:

```python
def get_db_for_user(user_id: str):
    """
    Repository Pattern: Returns a DB session scoped to a specific user.
    Uses SET LOCAL to scope the variable to the current transaction only,
    preventing cross-request contamination in the connection pool.
    """
    async def _get_db():
        async with async_session() as session:
            try:
                await session.execute(
                    text("SET LOCAL app.current_user_id = :uid"),
                    {"uid": user_id},
                )
                yield session
            finally:
                await session.close()
    return _get_db
```

Key design decisions:
- **`SET LOCAL`** (not `SET`) — scoped to the current transaction, so the variable is automatically cleared when the transaction ends, even in a pooled connection environment
- **Factory function** — `get_db_for_user(user_id)` returns a callable that FastAPI's `Depends()` can use as a dependency
- **Registered on `app.state`** — the factory is attached to `app.state.get_db_for_user` during lifespan startup, making it accessible in any router that receives a `Request` object, without creating a circular import

### 2.3 User-Scoped Endpoints (`services/analytics/routers/analytics_router.py`)

**Pattern: Repository Pattern + FastAPI Dependency Injection**

The analytics router was updated to use `get_db_for_user` for all user-facing endpoints:

1. **Shared auth dependency `_require_user_id`** — extracts and validates the `X-User-ID` header. This header is set by the nginx gateway after JWT verification, so the user is always authenticated at this point. This replaces the duplicate `if not x_user_id: raise HTTPException(...)` in every endpoint.

2. **`get_analytics_service` dependency** — retrieves `get_db_for_user` from `request.app.state`, creates a user-scoped session, and injects it into `AnalyticsService`. All queries inside the Facade (`_get_category_distribution`, `_get_budget_status`, etc.) now execute with RLS context set.

3. **`fhs/history` and `trends` endpoints** — these query the DB directly (not through `AnalyticsService`), so they also retrieve `get_db_for_user` from `app.state` and create a scoped session per-request.

```python
async def get_analytics_service(
    request: Request,
    user_id: str = Depends(_require_user_id),
    redis_client: aioredis.Redis = Depends(get_redis),
) -> AnalyticsService:
    get_db_for_user = request.app.state.get_db_for_user
    db_factory = get_db_for_user(user_id)
    db_gen = db_factory()
    db: AsyncSession = await db_gen.__anext__()
    # ... build AnalyticsService with scoped db session
```

**FHS Score Calculation Rationale (68.5/100):**
| Component | Metric | Value | Score (out of 25) |
|---|---|---|---|

| DTI Ratio | 15% | dti = 0.15 | 14.6 pts |

| Emergency Fund | 2 months | ef_months = 2 | 16.67 pts |

Score formula from `FHSProcessor.compute()`:
- `savings_rate` contributes: `min(rate / 0.20, 1) * 25`  
- `dti_ratio` contributes: `max(0, 1 - dti / 0.36) * 25`
- `spending_volatility` contributes: `max(0, 1 - cv / 0.5) * 25`
- `emergency_fund_months` contributes: `min(ef / 3, 1) * 25`

## 3. Design Patterns Used

### Repository Pattern (Backend)
`get_db_for_user()` encapsulates the data access concern of setting RLS context. Endpoint handlers don't need to know *how* user isolation is enforced — they just declare `db: AsyncSession = Depends(get_db_for_user_dependency)` and get a properly scoped session.

### Facade Pattern (existing, improved)
`AnalyticsService.get_dashboard_overview()` is already a Facade. This task improved it by ensuring the underlying queries receive RLS context, making the Facade actually work end-to-end.

### Null Object Pattern (Frontend)
Instead of conditional null-checks scattered throughout JSX, the `EmptyState` component acts as a Null Object — it renders a valid, complete UI state (with icon, title, description) when data is absent. Components switch between their real render and the `EmptyState` without additional branching at the widget level.

### Defensive Programming (Database)
The `nullif(current_setting(..., true), '')::UUID` pattern converts a potentially-crashing cast into a safe null comparison, following the principle of fail-safe defaults.
